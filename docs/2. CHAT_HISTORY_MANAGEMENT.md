# Chat History Memory Management - Redis & PostgreSQL

## Overview

The Document AI system now uses a **hybrid architecture** for chat history management, combining the speed of **Redis** (in-memory caching) with the durability of **PostgreSQL** (persistent storage).

---

## Architecture

### Dual-Storage System

```
┌─────────────────────────────────────────────────────────┐
│                   Chat Request                          │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
        ┌────────────────────────┐
        │  Hybrid Session        │
        │      Manager           │
        └────────┬───────────────┘
                 │
        ┌────────┴────────┐
        │                 │
        ▼                 ▼
┌──────────────┐  ┌──────────────┐
│    Redis     │  │  PostgreSQL  │
│   (Cache)    │  │ (Persistent) │
│              │  │              │
│ • Fast       │  │ • Durable    │
│ • Temporary  │  │ • Complete   │
│ • Active     │  │ • History    │
│   Sessions   │  │   Archive    │
└──────────────┘  └──────────────┘
```

---

## Components

### 1. **Redis Cache** (In-Memory, Fast Access)

**Purpose**: Store current active conversation sessions for ultra-fast access

**Features**:
- ⚡ Sub-millisecond response times
- 🔄 Auto-expiring sessions (TTL: 1 hour default)
- 💾 Stores recent N messages per session
- 🎯 Optimized for active users

**When to Use**:
- During active conversations
- Loading recent chat history
- Real-time message retrieval

**Data Structure**:
```json
{
  "session:tenant_id": "active",
  "messages:tenant_id": [
    {
      "role": "user",
      "content": "What is the contract about?",
      "timestamp": "2025-10-29T12:34:56Z"
    },
    {
      "role": "assistant",
      "content": "The contract is about...",
      "timestamp": "2025-10-29T12:34:58Z"
    }
  ]
}
```

---

### 2. **PostgreSQL Database** (Persistent, Complete History)

**Purpose**: Store complete chat history permanently

**Features**:
- 💾 Permanent storage
- 📊 Queryable history
- 🔍 Full-text search capability
- 📈 Analytics-ready
- 🔐 ACID compliant

**When to Use**:
- Historical queries
- Analytics and reporting
- Compliance and auditing
- Long-term data retention

**Database Schema**:

#### Table: `chat_messages`
| Column | Type | Description |
|--------|------|-------------|
| `id` | INTEGER | Primary key (auto-increment) |
| `tenant_id` | STRING(255) | Tenant/user identifier |
| `session_id` | STRING(255) | Session grouping |
| `role` | STRING(50) | 'user' or 'assistant' |
| `content` | TEXT | Message content |
| `timestamp` | DATETIME | Message timestamp (UTC) |

**Indexes**:
- `idx_tenant_timestamp` (tenant_id, timestamp)
- `idx_session_timestamp` (session_id, timestamp)

#### Table: `session_metadata`
| Column | Type | Description |
|--------|------|-------------|
| `session_id` | STRING(255) | Primary key |
| `tenant_id` | STRING(255) | Tenant identifier |
| `created_at` | DATETIME | Session start time |
| `last_activity` | DATETIME | Last message time |
| `message_count` | INTEGER | Total messages in session |

## Workflow

### 1. **Message Retrieval** (Read Path)

```
┌─────────────────┐
│ User sends      │
│ chat message    │
└────────┬────────┘
         │
         ▼
┌─────────────────────────┐
│ 1. Check Redis Cache    │
│    (Fast lookup)        │
└────────┬────────────────┘
         │
    ┌────┴────┐
    │ Found?  │
    └─┬────┬──┘
  YES │    │ NO
      │    │
      │    ▼
      │  ┌─────────────────────┐
      │  │ 2. Load from        │
      │  │    PostgreSQL       │
      │  └─────────┬───────────┘
      │            │
      │            ▼
      │  ┌─────────────────────┐
      │  │ 3. Cache in Redis   │
      │  └─────────┬───────────┘
      │            │
      └────────┬───┘
               │
               ▼
      ┌────────────────────┐
      │ 4. Return messages │
      │    to agent        │
      └────────────────────┘
```

**Performance**:
- ✅ **Cache HIT**: ~1-5ms (from Redis)
- ⏱️ **Cache MISS**: ~50-100ms (from PostgreSQL + cache update)

---

### 2. **Message Saving** (Write Path)

```
┌──────────────────┐
│ Agent generates  │
│ response         │
└────────┬─────────┘
         │
         ▼
┌─────────────────────────┐
│ 1. Save to PostgreSQL   │
│    (Persistent)         │
└────────┬────────────────┘
         │
         ▼
┌─────────────────────────┐
│ 2. Update Redis Cache   │
│    (Append message)     │
└────────┬────────────────┘
         │
         ▼
┌─────────────────────────┐
│ 3. Extend session TTL   │
│    (Keep active)        │
└────────┬────────────────┘
         │
         ▼
┌─────────────────────────┐
│ 4. Return success       │
└─────────────────────────┘
```

**Guarantees**:
- ✅ All messages saved to PostgreSQL (durable)
- ✅ Active sessions cached in Redis (fast)
- ✅ Automatic TTL management

---

### 3. **Session Expiry**

```
┌────────────────────┐
│ No activity for    │
│ 1 hour (TTL)       │
└────────┬───────────┘
         │
         ▼
┌────────────────────┐
│ Redis auto-evicts  │
│ from cache         │
└────────┬───────────┘
         │
         ▼
┌────────────────────┐
│ PostgreSQL retains │
│ full history       │
└────────────────────┘
```

**What happens**:
- ❌ Session removed from Redis (frees memory)
- ✅ History preserved in PostgreSQL (forever)
- 🔄 Next request loads from PostgreSQL → caches in Redis

## Configuration Options

### Redis TTL (Time-To-Live)

Control how long sessions stay in cache:

```env
# .env
REDIS_SESSION_TTL=3600  # 1 hour (default)
# REDIS_SESSION_TTL=7200  # 2 hours
# REDIS_SESSION_TTL=1800  # 30 minutes
```

**Recommendation**:
- **High traffic**: Shorter TTL (1800s / 30min) to free memory faster
- **Low traffic**: Longer TTL (7200s / 2hr) for better cache hits

---

### Message Limit

Control how many recent messages to load:

```python
# In hybrid_session_manager.py
db_messages = self.postgres.get_recent_messages(tenant_id, limit=50)  # Default: 50
```

**Trade-offs**:
- **Higher limit**: More context, slower initial load
- **Lower limit**: Less context, faster load

## Best Practices

### 1. Cache Hit Optimization

✅ **DO**:
- Keep TTL reasonable (1-2 hours)
- Use consistent tenant_id format
- Let cache warm up naturally

❌ **DON'T**:
- Force cache refreshes unnecessarily
- Use extremely short TTLs (< 5 minutes)
- Store huge message limits in cache

---

### 2. Database Maintenance

✅ **DO**:
- Regular backups of PostgreSQL
- Monitor database size
- Add indexes for custom queries

❌ **DON'T**:
- Store non-message data in chat tables
- Delete messages without proper cleanup
- Ignore slow query warnings

---

### 3. Session Management

✅ **DO**:
- Use `clear_session` for privacy (clears cache only)
- Use `delete_tenant` for GDPR compliance (deletes everything)
- Monitor active sessions via Redis

❌ **DON'T**:
- Manually delete from Redis/PostgreSQL
- Mix session_id and tenant_id concepts
- Assume cache always has data

## Troubleshooting

### Redis Connection Issues

**Problem**: `redis.ConnectionError`

**Solutions**:
1. Check Redis is running: `redis-cli ping`
2. Verify `.env` settings (host, port, password)
3. Check firewall rules

---

### PostgreSQL Connection Issues

**Problem**: `psycopg2.OperationalError`

**Solutions**:
1. Check PostgreSQL is running: `pg_isready`
2. Verify database exists: `psql -l`
3. Check credentials in `.env`
4. Ensure database `document_ai` is created

---

### Missing Messages

**Problem**: Messages not appearing in chat history

**Possible Causes**:
1. **Cache expired**: Normal behavior, will reload from PostgreSQL
2. **Database write failed**: Check PostgreSQL logs
3. **Wrong tenant_id**: Verify tenant_id matches

**Solution**:
```bash
# Check if in PostgreSQL
python scripts/check_dependencies.py

# View logs
tail -f logs/app.log
```

## Architecture Benefits

### ✅ **Performance**
- Fast access via Redis for active users
- No database query for every message

### ✅ **Reliability**
- Messages never lost (PostgreSQL persistence)
- Survives server restarts

### ✅ **Scalability**
- Redis handles thousands of concurrent sessions
- PostgreSQL can store millions of messages

### ✅ **Compliance**
- Complete audit trail in PostgreSQL
- GDPR-compliant deletion

### ✅ **Cost-Effective**
- Redis auto-evicts inactive sessions (saves memory)
- PostgreSQL compresses old data
